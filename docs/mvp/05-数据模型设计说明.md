# 05-数据模型设计说明

## 1. 核心建模原则

数据模型遵守：

- 所有租户业务数据带 `tenant_id`。
- 资产核心字段稳定，业务字段可配置。
- 有形资产和无形资产统一挂在 `asset` 主对象下。
- 动态字段默认使用 JSONB 保存。
- 热点查询字段、排序字段、唯一字段使用物理列或表达式/生成列索引。
- 生命周期、审批、附件、审计独立建模。
- 关键业务表使用软删除时，唯一约束必须采用 `WHERE deleted = false` 的部分唯一索引。

## 2. 主要实体

平台与身份：

- Tenant：租户。
- User：租户用户。
- Organization：组织。
- Role：角色。
- Permission：权限。

资产元数据：

- AssetType：资产类型。
- FieldDefinition：字段定义。
- FormSchema：表单配置。
- ListViewSchema：列表配置。
- SearchSchema：查询配置。

资产核心：

- Asset：资产主对象。
- AssetRelation：资产关系。
- Location：位置。
- AssetLocationHistory：位置历史。

生命周期：

- LifecycleTemplate：生命周期模板。
- LifecycleState：生命周期状态。
- LifecycleTransition：生命周期流转。
- LifecycleEvent：生命周期事件。

审批：

- ApprovalTemplate：审批模板。
- ApprovalNode：审批节点。
- ApprovalInstance：审批实例。
- ApprovalTask：审批任务。

采购：

- PurchaseRequest：采购申请。
- PurchaseItem：采购明细。

软件许可证：

- SoftwareLicense：软件许可证专属信息。
- LicenseAssignment：授权分配记录。

轻量资产：

- CertificateRecord：证书记录。
- DomainRecord：域名记录。
- CloudResourceRecord：云资源记录。

运营：

- Attachment：附件。
- Notification：通知。
- AuditLog：审计日志。
- ImportJob：导入任务。
- WebhookSubscription：Webhook 订阅。
- ApiToken：外部 API Token。

## 3. 资产统一模型

`Asset` 代表所有资产。

字段：

- `asset_kind` 区分有形和无形。
- `asset_type_id` 决定字段、表单、列表、查询和生命周期模板。
- `lifecycle_status` 保存当前状态。
- `attributes` 保存非热点动态字段。

必须落为物理列的热点字段：

```text
asset_no
asset_name
asset_kind
asset_type_id
lifecycle_status
owner_user_id
owner_org_id
responsible_user_id
location_id
cost_center_id
serial_no
brand
model
vendor
warranty_end_date
license_end_date
```

说明：

- `serial_no` 对应 SN/序列号，不能只放在 JSONB。
- `license_end_date`、`warranty_end_date` 用于到期提醒和范围查询。
- 证书、域名的到期日期应在专属表中保留物理列，不只放在 JSONB。
- 其他低频、展示型、扩展型字段保留在 `attributes`。

服务器资产示例：

```json
{
  "assetNo": "SRV-2026-0001",
  "assetName": "核心数据库服务器01",
  "brand": "Dell",
  "model": "R750",
  "serialNo": "SN123456",
  "attributes": {
    "memory_gb": 512,
    "cpu_spec": "2 x Xeon"
  }
}
```

软件许可证示例：

```json
{
  "assetNo": "LIC-2026-0001",
  "assetName": "数据库授权",
  "vendor": "Microsoft",
  "licenseEndDate": "2026-12-31",
  "attributes": {
    "license_model": "volume"
  }
}
```

## 4. JSONB 字段、索引与唯一性策略

动态字段处理规则：

| 字段类型 | 存储策略 | 约束策略 |
|---|---|---|
| 高频筛选/排序字段 | 物理列 | 普通 BTree 索引 |
| 唯一字段 | 物理列优先 | 部分唯一索引 |
| 敏感字段 | 加密列或加密 JSON 值 | 后端脱敏输出 |
| 低频扩展字段 | `attributes` JSONB | GIN 索引或无索引 |
| 范围查询日期 | 物理列 | BTree 索引 |

`field_definitions.unique_scope` 不能只靠应用层判断。实现必须按以下优先级处理：

1. 字段在热点白名单内：使用物理列和部分唯一索引。
2. 字段不在白名单但声明唯一：为该字段生成表达式唯一索引，或拒绝发布该唯一配置并提示需要升格为热点字段。
3. 写入时应用层仍需在同一事务内做重复检查，用于返回友好错误信息；数据库唯一约束是最终保证。

## 5. 关系模型

资产关系用于表达：

- 服务器安装软件许可证。
- 域名绑定证书。
- 域名指向云资源。
- 云资源属于业务系统。
- 设备位于机柜或 U 位。

关系由 `asset_relation` 保存，不把关系写死在某个资产类型中。

## 6. 生命周期模型

每个资产类型可以绑定一个生命周期模板。

资产只保存当前状态，历史流转写入 `lifecycle_event`。

审批不是生命周期本身，而是生命周期动作的前置条件。

MVP-2 不依赖审批即可完成状态机验收；MVP-3 开始通过 `approval_instance.action_context` 绑定待执行生命周期动作。

## 7. 权限模型

权限由五层组成：

```text
租户隔离
功能权限
数据范围权限
字段权限
状态权限
```

字段权限必须影响：

- 页面详情。
- 列表列。
- 表单编辑。
- 导出。
- API 响应。

## 8. 采购与资产创建关系

采购申请是独立业务对象，不是资产创建的强制前置。

MVP-1：

- 资产可由资产管理员手工创建。
- 初始状态默认为资产类型生命周期模板的起始状态。

MVP-2/MVP-3：

- 采购申请可触发生命周期动作。
- 采购审批通过后可创建资产草稿或推动既有资产进入 `purchasing`。
- 采购记录与资产通过 `purchase_items.asset_id` 关联，允许一个采购申请关联多个资产。

## 9. 数据一致性边界

必须使用同一数据库事务保证：

- 资产状态更新和 `lifecycle_events` 写入。
- 审批通过回调和生命周期流转。
- 软件许可证分配：`license_assignment` 新增与 `software_license.used_quantity` 更新。
- 资产软删除和关系、附件可见性更新。

并发控制：

- 许可证分配使用行级锁或乐观锁，避免超额分配。
- 资产编号和序列号使用数据库唯一索引兜底。
- 生命周期动作执行时校验当前状态；状态已变化返回 `409 CONFLICT`。
