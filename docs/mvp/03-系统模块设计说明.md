# 03-系统模块设计说明

## 1. 总体架构

系统采用模块化单体架构。

部署形态上第一版是单体，代码结构上按业务域拆分，后续可按模块边界拆服务。

后端分层：

```text
controller -> application service -> domain service -> repository -> database
```

模块之间通过 application service 或领域事件交互，不允许跨模块直接改对方表的核心状态。

## 2. 模块清单

| 模块 | 职责 |
|---|---|
| platform | 平台租户管理、平台参数、平台审计 |
| identity | 用户、组织、角色、认证 |
| permission | 功能权限、数据权限、字段权限、状态权限 |
| metadata | 资产类型、字段、表单、列表、查询配置 |
| asset-core | 资产主对象、动态字段值、资产关系 |
| lifecycle | 生命周期模板、状态、流转、事件 |
| approval | 审批模板、节点、实例、任务 |
| procurement | 采购申请和采购记录 |
| inventory | 入库、登记、启用、回收 |
| maintenance | 故障、维修、维保 |
| license | 软件许可证、授权分配、回收、续费 |
| certificate | 证书和域名轻量管理 |
| cloud-resource | 云资源轻量登记、成本归属、同步接口预留 |
| notification | 站内通知、提醒、邮件和 Webhook 预留 |
| audit | 审计日志 |
| attachment | 附件、合同、授权文件、验收单 |
| report | 看板和基础报表 |
| integration | API Token、Webhook、外部接口 |

## 3. 模块边界

### 3.1 asset-core

只负责资产共同能力：

- 资产主对象。
- 动态属性。
- 资产关系。
- 通用查询。

不负责：

- 许可证数量计算。
- 生命周期审批。
- 维修业务。
- 报表聚合。

资产的 `lifecycle_status` 只能由 `lifecycle` 模块通过动作修改。

### 3.2 metadata

负责字段、表单、列表、查询元数据。

动态字段定义属于 `metadata`，动态字段值属于 `asset-core`。

元数据模块必须维护字段是否为热点字段、是否唯一、是否敏感、是否加密，并把这些规则提供给资产写入、查询、导出和权限过滤使用。

### 3.3 lifecycle

负责状态机和状态流转。

资产状态不能由 `asset-core` 直接改，只能由 `lifecycle` 动作改变。

MVP-2 只实现状态机、动作、守卫和事件日志，不依赖审批模块即可验收。

MVP-3 起，`lifecycle` 在执行动作前判断 transition 是否需要审批：

1. 不需要审批：校验守卫规则后直接流转。
2. 需要审批：创建审批实例和任务，保存待执行动作上下文，资产状态保持不变。
3. 审批通过：由审批模块回调生命周期模块，重新校验状态和守卫后执行流转。
4. 审批驳回或撤销：资产状态保持不变，写审批事件和通知。

### 3.4 approval

负责审批模板、审批节点、审批实例和审批任务，不直接决定资产业务规则。

审批实例必须关联业务对象和动作上下文：

- `ref_type`：如 `asset`、`purchase_request`、`license_assignment`。
- `ref_id`：业务对象 ID。
- `action_code`：生命周期动作编码。
- `action_context`：动作请求参数、附件、原因等 JSON。

审批通过后只能通过生命周期模块公开回调执行状态流转，不能直接更新资产状态。

### 3.5 permission

权限由后端统一执行，不能散落在页面或业务服务中。

权限模块提供：

- 功能权限校验。
- 数据范围条件注入。
- 字段可见、可编辑、脱敏、可导出过滤。
- 状态动作可执行矩阵。

字段权限基础能力从 MVP-1 开始实现；MVP-3 在此基础上补齐审批、状态权限和通知联动。

### 3.6 procurement

采购申请是独立业务对象，可关联一个或多个目标资产。

MVP-1 允许资产独立创建；采购模块不作为资产创建的强制前置。

采购通过后可触发生命周期动作，例如 `submit_purchase` 或 `confirm_inbound`，并写入采购记录和生命周期事件。

## 4. 运行链路

```text
请求
-> 认证过滤器
-> 平台/租户上下文识别
-> 功能权限检查
-> 数据权限注入
-> Controller
-> Application Service
-> Domain Service
-> Repository
-> 审计和通知
-> 字段权限过滤
-> 响应
```

平台域接口和租户域接口必须在路由或拦截器层区分。平台用户不得获得租户上下文；租户用户不得调用平台域管理接口。

## 5. 前端模块

```text
src/modules/
  tenant/
  identity/
  permission/
  metadata/
  asset/
  lifecycle/
  approval/
  license/
  certificate/
  cloudresource/
  report/
  integration/
```

前端只负责展示和交互。权限判断以后端返回的权限码、字段权限、状态动作权限为准。

前端工程必须遵守 `16-UI设计规范与前端交互范式.md`，同时建立统一 API client、错误处理、权限指令、表格/筛选/抽屉/动态表单二次封装。
