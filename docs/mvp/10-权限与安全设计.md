# 10-权限与安全设计

## 1. 权限模型

权限采用五层模型：

```text
租户隔离
功能权限
数据范围权限
字段权限
状态权限
```

权限必须在后端执行，前端只用于改善交互体验。

## 2. 租户隔离与上下文边界

第一版采用共享数据库、共享表、`tenant_id` 隔离。

规则：

- 前端不能传可信 `tenant_id`。
- 后端从 token 中解析租户上下文。
- 所有租户业务表必须有 `tenant_id`。
- 查询自动注入租户过滤。
- 写入自动填充租户。
- 缓存 key 必须包含租户 ID。
- 附件路径必须包含租户隔离目录。

平台侧边界：

- `platform_admin`、`platform_auditor` 无租户上下文。
- 平台角色只能访问 `/api/v1/platform/**` 和平台健康、平台审计类接口。
- 平台角色不能直接访问租户资产、审批、报表、附件。
- 租户用户不能访问平台租户管理接口。

租户用户边界：

- 一个登录用户可以属于多个租户，但每个 access token 只能绑定一个当前租户上下文。
- `switch-tenant` 只能切换到用户已加入且启用的租户。
- 服务端必须校验用户、角色、组织均属于当前租户。

API Token 边界：

- API Token 必须绑定一个 `tenant_id`。
- API Token 必须绑定 scopes。
- API Token 数据库只保存哈希，明文只在创建时返回一次。
- API Token 不能调用平台域接口。

## 3. 角色

平台角色：

- `platform_admin`
- `platform_auditor`

租户角色：

- `tenant_admin`
- `asset_admin`
- `asset_operator`
- `license_admin`
- `procurement_user`
- `maintenance_user`
- `department_manager`
- `asset_user`
- `auditor`
- `approver`

`approver` 是职能角色。是否能审批某个任务由审批任务归属、审批节点规则、数据范围和任务状态共同决定。

## 5. 角色-权限缺省绑定矩阵

| 角色 | 默认功能权限 | 默认数据范围 | 默认字段权限 | 备注 |
|---|---|---|---|---|
| platform_admin | 平台租户管理、平台状态、平台审计 | 平台域 | 不访问租户业务字段 | 无租户上下文 |
| platform_auditor | 平台审计只读 | 平台域 | 不访问租户业务字段 | 无租户上下文 |
| tenant_admin | identity/metadata/permission 全量、asset/report/audit 查看 | all_tenant | 默认全字段可见，可编辑非系统字段 | 租户内最高管理员 |
| asset_admin | asset 全量、lifecycle、import/export、report | all_tenant | 资产字段可见可编辑，敏感字段按规则脱敏 | 不管理用户权限 |
| asset_operator | asset 查看更新、lifecycle、maintenance | specified_location + specified_asset_type | 运维字段可编辑，财务/合同字段不可见或脱敏 | 机房运维 |
| license_admin | license 全量、软件资产 lifecycle、report | specified_asset_type=software_license | 许可证字段可见，license key 默认脱敏，可按授权查看 | 不能处置机房设备 |
| procurement_user | procurement 查看提交、asset 查看 | own_org | 采购字段可编辑，资产核心字段只读 | 采购申请人 |
| maintenance_user | maintenance 管理、asset 查看、故障动作 | specified_location | 维修字段可编辑，财务/合同字段不可见 | 维修人员 |
| department_manager | asset 查看、approval 查看处理、report 查看 | own_org_tree | 资产核心字段可见，敏感字段脱敏，通常不可编辑 | 部门审批 |
| asset_user | asset 查看、个人待办 | self/responsible | 只看本人相关字段，敏感字段隐藏 | 普通使用人 |
| auditor | audit/report/asset 只读 | all_tenant | 字段只读，敏感字段脱敏，不可导出敏感字段 | 审计员 |
| approver | approval:view/approve/reject | 由审批任务和节点决定 | 审批上下文字段可见，敏感字段按业务规则脱敏 | 职能角色 |

默认矩阵是初始化基线，租户管理员可在权限配置中收紧或扩展，但不能突破平台/租户边界。

## 6. 数据范围权限

范围类型：

```text
self
own_org
own_org_tree
specified_org
specified_location
specified_asset_type
responsible
all_tenant
```

示例：

- 普通员工：`self`。
- 部门负责人：`own_org_tree`。
- 机房运维：`specified_location + specified_asset_type`。
- 软件管理员：`specified_asset_type = software_license`。
- 租户管理员：`all_tenant`。

数据范围必须应用于列表、详情、导出、报表和审批任务查询。

## 7. 字段权限

字段权限控制：

- 是否可见。
- 是否可编辑。
- 是否脱敏。
- 是否可导出。

字段权限必须作用于：

- 详情页。
- 列表页。
- 表单。
- API 响应。
- 导出。

敏感字段：

- 软件授权密钥。
- 采购金额。
- 合同附件。
- 云资源密钥类信息。

字段权限基础从 MVP-1 开始实现；MVP-3 增强为可配置规则。

脱敏规则示例：

- `license_key`：只显示后 4 位，其余为 `***`。
- `purchase_amount`：无权限返回 `null` 或 `***`。
- 合同附件：无权限不返回下载地址。

## 8. 状态权限

状态权限控制当前生命周期状态下允许的动作。

状态权限落地为 `state_permission_rules(role_id, asset_type_id, lifecycle_status, action_code, allowed)`。

示例：

- 退役资产不可编辑核心字段。
- 普通用户不能执行退役。
- 软件管理员能分配许可证，但不能处置机房设备。
- 审批中资产申请人不能修改关键字段。

状态权限与功能权限关系：

- 功能权限决定用户是否具备某类能力，例如 `lifecycle:transition`。
- 状态权限决定用户在某类资产某个状态下能不能执行某个动作。
- 两者必须同时满足。

## 9. 审批权限

审批必须满足：

```text
当前用户是审批任务审批人
或 当前用户符合审批节点规则
且 业务对象在当前用户数据范围内
且 审批任务处于 pending
```

非审批人不能通过接口越权审批。

审批通过不直接修改资产状态，必须通过生命周期模块回调执行 transition。

## 10. 安全设计

认证：

- Access token 短期有效。
- Refresh token 必须轮换，旧 refresh token 使用后失效。
- 退出登录撤销 refresh token。
- 初始用户必须设置 `must_change_password=true`，首次登录只允许改密。

密码：

- 密码哈希使用 BCrypt/Argon2。
- 默认密码不得用于生产。
- 连续失败登录应锁定或限流。

敏感字段：

- `field_definitions.encrypted=true` 的字段必须加密存储。
- 软件许可证密钥使用加密列或加密 JSON 值，不能明文保存。
- 日志、审计、导出不得输出敏感明文。

Webhook：

- 投递必须使用 HMAC-SHA256 签名。
- 签名头包含事件、时间戳和签名。
- 接收方可按时间戳防重放。
- 投递失败写日志并按配置重试。

缓存：

- 权限、元数据缓存 key 必须包含租户 ID。
- 权限变更、角色变更、字段变更、资产类型变更必须触发对应租户缓存失效。

## 11. 审计

必须审计：

- 登录。
- 角色授权。
- 权限规则变更。
- 元数据变更。
- 资产新增、编辑、删除。
- 生命周期动作。
- 审批通过、驳回、撤销、重新提交。
- 导入、导出。
- 附件下载。
- API Token 创建和撤销。
- Webhook 配置变更。

审计日志必须记录：

```text
tenant_id
operator_id
action
resource_type
resource_id
before_data
after_data
ip
user_agent
created_at
```
