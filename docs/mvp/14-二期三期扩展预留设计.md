# 14-二期三期扩展预留设计

## 1. 扩展原则

第一版只实现 MVP 必需能力，但底层必须预留扩展点。

不得为了扩展点提前实现复杂功能。

## 2. 二期建议范围

二期可扩展：

- 办公终端完整生命周期。
- 备件/耗材库存管理。
- 合同管理。
- 供应商管理。
- 软件许可证合规增强。
- 云资源同步接口实现。
- 邮件通知。
- Webhook 投递重试增强和失败告警。
- 更细粒度报表。
- 前端暗色主题和高密度模式。
- 核心复杂页面线框图升级为可交互原型。

## 3. 三期建议范围

三期可扩展：

- CMDB 拓扑。
- 自动资产发现。
- 云厂商自动同步。
- 工单系统集成。
- 完整流程设计器。
- 条件权限可视化编辑器。
- 多租户独立 schema 或独立数据库。
- SaaS 计费和套餐。
- 移动端适配。

## 4. 数据模型预留

预留字段：

- `source_type`：manual / import / api / discovery。
- `sync_source`：云资源同步来源。
- `attributes`：动态字段扩展。
- `condition_rule`：流程和权限条件扩展。
- `metadata_version`：元数据发布版本。

## 5. 接口预留

预留接口：

```text
POST /api/v1/integrations/cloud-sync/jobs
GET  /api/v1/integrations/cloud-sync/jobs/{jobId}
POST /api/v1/integrations/discovery/jobs
GET  /api/v1/integrations/discovery/jobs/{jobId}
```

第一版可以不实现这些接口，只保留设计说明。

## 6. 架构演进

第一版模块化单体。

后续可拆分：

- identity 服务。
- asset 服务。
- workflow 服务。
- integration 服务。
- report 服务。

拆分条件：

- 模块边界稳定。
- 数据归属清晰。
- 接口稳定。
- 性能或部署独立性有明确需求。

## 7. 租户隔离演进

第一版：

```text
共享库 + 共享表 + tenant_id
```

后续可演进：

```text
共享库 + 独立 schema
独立数据库
```

前提：

- 所有数据访问都经过租户上下文。
- 不在业务代码中散落租户判断。
- 缓存、附件、任务都带租户隔离。

## 8. RLS 纵深防御预留

PostgreSQL Row Level Security 不作为 MVP 阻塞项，但应作为二期安全加固方案评估。

建议演进路径：

1. MVP 先确保所有 repository 查询自动注入 `tenant_id`。
2. 建立跨租户访问自动化测试。
3. 二期在核心租户业务表试点 RLS 策略。
4. RLS 策略稳定后推广到资产、审批、附件、审计等表。

RLS 是纵深防御，不替代应用层权限、数据范围和字段权限。
