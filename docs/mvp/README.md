# IT资产管理平台 MVP 开发材料说明

## 1. 目录用途

本目录用于交付给 AI 开发平台或软件开发团队，作为“IT资产管理平台”从 0 到 1 开发的正式依据。

本系统不是旧系统改写，而是重新设计的新一代多租户、元数据驱动、生命周期优先的 IT 资产管理平台。

第一版定位为“单企业内多组织、多地点、多资产类型”的企业级 ITAM 平台，同时从底层预留 SaaS 多租户能力。第一版不做 SaaS 计费和租户自助开户注册。

第一阶段目标是完成可运行的 ITAM MVP，并按阶段推进：

```text
MVP-0 平台底座
MVP-1 元数据与资产核心
MVP-2 生命周期状态机与动作（不含审批）
MVP-3 审批、权限增强、通知
MVP-4 导入导出、报表、轻量资产扩展
MVP-5 生产化加固与集成
```

## 3. 已确认技术栈

```text
后端：Spring Boot 3 + Java 21
数据库：PostgreSQL
缓存：Redis
前端：Vue 3 + TypeScript + Vite
UI：Element Plus
数据库迁移：Flyway
API 文档：OpenAPI / Swagger
认证：JWT + Refresh Token
部署：Docker Compose 起步，后续可演进 Kubernetes
```

开发平台不得自行切换技术栈。若因平台限制必须调整，必须保持功能、数据模型、API 语义、权限规则和验收标准一致。

## 4. 第一版必须实现

第一版必须实现：

- 多租户平台底座。
- 用户、组织、角色、权限。
- 元数据驱动资产类型、字段、表单、列表、查询。
- 有形资产中的机房资产完整生命周期样板。
- 无形资产中的软件许可证完整生命周期样板。
- 证书、域名、云资源轻量管理。
- 生命周期状态机和动作 API。
- 轻量审批，并与生命周期动作联动。
- 数据权限、字段权限、状态权限。
- 导入导出、基础报表、提醒、审计。
- API Token、Webhook 基础、附件权限、敏感字段控制。

## 5. 第一版明确不实现

第一版不实现：

- 旧系统历史数据迁移。
- 完整低代码平台。
- BPMN 流程设计器。
- 云资源自动发现和自动同步。
- 复杂 CMDB 拓扑图。
- 资产自动扫描。
- 每租户独立数据库。
- 全量 BI 报表设计器。
- SaaS 计费。

## 6. 已确认的评审修订决策

- MVP-2 更名为“生命周期状态机与动作（不含审批）”。MVP-2 验证状态机、动作、守卫和事件日志；从 MVP-3 开始叠加审批。
- MVP-2 中标记为 `require_approval=true` 的动作可按“免审批执行”或“返回 approval_required 且不改状态”两种模式配置。第一版开发建议采用“免审批执行”便于独立验收；MVP-3 后统一创建审批实例。
- 字段权限基础能力前置到 MVP-1，确保动态表单、列表、详情和导出从第一版核心资产阶段就能按权限渲染。
- 动态字段默认进入 `assets.attributes` JSONB，但热点字段和唯一字段必须有物理列或数据库索引策略，不允许只靠应用层校验。
- 平台用户无租户上下文，只能访问平台域接口；租户用户必须在已授权租户上下文内访问租户业务接口。
- API Token 必须绑定租户和 scopes，只存哈希，创建时仅返回一次明文。
- `approver` 是职能角色，不代表任意审批权。审批权必须同时满足任务归属、节点规则、数据范围和任务状态。
- 采购申请是独立业务对象，资产可以在 MVP-1 独立创建；采购流程在后续动作中作为资产来源和生命周期前置记录。

## 7. 最小演示主线

第一版必须保证以下主线可运行：

```text
平台管理员创建租户
-> 租户管理员创建组织、用户、角色
-> 租户管理员配置资产类型、字段、表单、列表
-> 资产管理员创建机房资产
-> 机房资产完成采购、入库、上架、运行、维修、退役、处置
-> 软件管理员创建软件许可证
-> 软件许可证完成采购、授权登记、分配、回收、续费、停用
-> 普通用户和部门负责人按权限查看资产
-> 审批人处理采购、退役、软件授权分配审批
-> 系统记录审计日志、通知和报表
```

## 8. 开发约束

开发时必须遵守：

- 租户数据必须通过 `tenant_id` 隔离。
- 前端传入的 `tenant_id` 不可信。
- 资产状态不能直接编辑，只能通过生命周期动作 API 改变。
- 动态字段由元数据驱动，不能写死为单一页面。
- 字段权限必须在后端生效，不能只靠前端隐藏。
- 导出必须遵守字段权限。
- 关键写操作必须写审计日志。
- 不得提前实现第一版以外的复杂能力。
