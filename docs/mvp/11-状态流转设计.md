# 11-状态流转设计

## 1. 状态流转原则

资产状态只能通过生命周期动作改变。

禁止：

- 前端直接修改 `lifecycle_status`。
- 资产编辑接口直接修改生命周期状态。
- 跳过守卫规则强制改状态。

MVP-2 是“生命周期状态机与动作（不含审批）”，用于验证状态、动作、守卫和事件日志。MVP-3 开始把审批作为动作前置条件叠加到状态机上。

## 2. 机房资产状态

状态：

```text
planned        规划/申请
purchasing     采购中
inbound        已入库
deployed       已上架/已部署
running        运行中
maintenance    维修中
retired        已退役
disposed       已处置
```

流转：

```text
planned -> purchasing      submit_purchase
purchasing -> inbound      confirm_inbound
inbound -> deployed        deploy
deployed -> running        start_operation
running -> maintenance     report_fault
maintenance -> running     finish_repair
maintenance -> retired     repair_failed_retire
running -> retired         retire
retired -> disposed        dispose
```

审批要求：

- `submit_purchase` 需要审批，从 MVP-3 开始生效。
- `retire` 需要审批，从 MVP-3 开始生效。
- `dispose` 需要附件和审计，可按租户策略要求审批。

MVP-2 中上述动作可免审批执行，以便状态机独立验收。

## 3. 软件许可证状态

状态：

```text
planned
purchasing
available
assigned
renewal_pending
expired
reclaimed
retired
```

流转：

```text
planned -> purchasing          submit_purchase
purchasing -> available        register_license
available -> assigned          assign_license
assigned -> reclaimed          reclaim_license
assigned -> renewal_pending    mark_renewal
renewal_pending -> assigned    renew_license
renewal_pending -> expired     expire
expired -> assigned            renew_after_expired
available -> retired           retire
reclaimed -> available         release_to_pool
```

审批要求：

- `submit_purchase` 需要审批，从 MVP-3 开始生效。
- `assign_license` 可按策略审批。
- `renew_license` 可按策略审批。

## 4. 证书/域名状态

状态：

```text
active
renewal_pending
expired
retired
```

第一版不做复杂流程，只做状态维护和到期提醒。

## 5. 云资源状态

状态：

```text
planned
active
suspended
retired
```

第一版手工登记和维护，后续预留云同步。

## 6. 守卫规则

守卫规则支持：

- 必填字段。
- 必填动态字段。
- 必填附件。
- 必要权限。
- 可用数量。
- 当前状态。
- 数据范围。
- 字段可编辑性。

示例：

服务器 `deploy` 必须满足：

- 已填写位置。
- 已填写责任人。
- 已填写 SN。
- 当前状态为 `inbound`。
- 当前用户有 `lifecycle:transition` 权限。
- 当前用户在该资产类型和状态下允许执行 `deploy`。

许可证 `assign_license` 必须满足：

- 可用授权数量大于等于分配数量。
- 分配对象合法。
- 当前用户有 `license:assign` 权限。
- 并发分配时不能超过 `total_quantity`。

## 7. 生命周期与审批联动

每个 `lifecycle_transition` 可配置：

```text
require_approval
approval_template_id
guard_rule
```

动作接口执行规则：

1. 校验当前租户、功能权限、数据范围、状态权限。
2. 校验当前状态是否等于 transition 的 `from_state`。
3. 校验守卫规则。
4. 如果不需要审批：更新资产状态，写 `lifecycle_events`，发送通知。
5. 如果需要审批：创建 `approval_instance` 和首个 `approval_task`，保存 `action_context`，资产状态保持不变。
6. 审批通过：审批模块调用生命周期服务，重新校验当前状态后执行 transition。
7. 审批驳回：审批实例为 `rejected`，资产状态不变，写通知。
8. 审批撤销：审批实例为 `cancelled`，资产状态不变，写通知。
9. 重新提交：基于原动作上下文创建新审批实例，原实例不复用。

审批映射示例：

| 动作 | 审批模板 | 节点 |
|---|---|---|
| `submit_purchase` | `purchase_approval` | 部门负责人 -> 资产管理员 |
| `retire` | `asset_retire_approval` | 部门负责人 -> 审计员 |
| `assign_license` | `license_assign_approval` | 部门负责人 |
| `renew_license` | `license_renew_approval` | 软件管理员负责人 |

## 8. 异常分支

必须支持：

- 采购申请撤销：`purchase_request.status = cancelled`，资产状态不变或回到原状态。
- 审批驳回：资产状态不变，允许重新提交。
- 维修失败：`maintenance -> retired`。
- 许可证过期后续费：`expired -> assigned`。
- 非法流转：返回 `422 BUSINESS_RULE_VIOLATION`。
- 并发流转：若当前状态已变化，返回 `409 CONFLICT`。
- 处置缺附件：返回 `422 BUSINESS_RULE_VIOLATION`。

## 9. 事件记录

每次成功流转写入：

- 资产 ID。
- 原状态。
- 新状态。
- 动作编码。
- 操作人。
- 原因。
- 动作表单数据。
- 附件 ID。
- 关联审批实例。
- 创建时间。

审批创建、审批通过、驳回、撤销也必须写审计日志；只有真正改变资产状态的动作写 `lifecycle_events`。
