# UI 规范长期要求

以后所有带前端界面的软件系统开发材料，必须包含 `16-UI设计规范与前端交互范式.md`。该文档用于定义视觉风格、布局范式、组件规则、页面状态、响应式要求和 UI 验收标准。

# 软件系统开发材料生成标准

## 1. 适用范围

以后所有软件系统开发类项目，在进入开发平台或交付给外部开发团队前，必须按本标准生成材料。

本标准适用于：

- 新系统从 0 到 1 建设。
- 旧系统重构为新系统。
- MVP 分期开发。
- 交给 AI 开发平台、外包团队或内部研发团队实现。

## 2. 标准目录

所有项目必须在项目工作目录下生成：

```text
docs/mvp/
  01-产品需求说明书_PRD.md
  02-MVP功能清单与用户故事.md
  03-系统模块设计说明.md
  04-页面清单与交互说明.md
  05-数据模型设计说明.md
  06-数据库表结构设计.md
  07-API接口规格说明.md
  08-模板体系设计.md
  09-报告生成规则说明.md
  10-权限与安全设计.md
  11-状态流转设计.md
  12-测试用例与验收标准.md
  13-种子数据与演示数据.md
  14-二期三期扩展预留设计.md
  15-开发总提示词.md
  16-UI设计规范与前端交互范式.md
  README.md
```

如果项目较复杂，可以额外保留过程文档：

```text
docs/superpowers/specs/
docs/superpowers/plans/
docs/handoff/
```

但正式交付给开发平台时，入口必须是 `docs/mvp/README.md`。

## 3. 文档职责

每份文档职责固定：

- `01` 定义产品目标、范围、角色、业务闭环、边界。
- `02` 定义 MVP 功能清单和用户故事。
- `03` 定义系统模块、模块边界、模块依赖。
- `04` 定义页面、交互、字段、操作入口。
- `05` 定义业务实体和实体关系。
- `06` 定义数据库表、字段、索引、约束。
- `07` 定义 API 路径、请求、响应、错误码。
- `08` 定义模板、元数据、可配置字段、表单、列表。
- `09` 定义报表、看板、生成规则和数据来源。
- `10` 定义认证、权限、租户隔离、安全、审计。
- `11` 定义状态机、流程动作、审批联动。
- `12` 定义测试用例、验收标准、质量门禁。
- `13` 定义初始化数据、演示数据、默认账号。
- `14` 定义后续阶段扩展边界和预留点。
- `15` 定义可直接交给 AI 开发平台的总提示词。
- `16` 定义 UI 设计规范、前端交互范式、组件规则、关键页结构和前端工程约束。
- `README` 定义阅读顺序、范围摘要、开发约束。

## 4. 生成原则

生成材料时必须遵守：

- 先确认范围和边界，再生成开发材料。
- 不把需求、设计、开发计划混成一个大文档。
- 每个文档都必须可单独阅读。
- 表、接口、状态、权限必须使用稳定编码。
- MVP 必须明确做什么和不做什么。
- 不使用任何未完成标记或空泛补充说明。
- 不在第一期实现二期、三期能力，只保留扩展点。
- 所有验收标准必须可测试。
- 对外部开发平台，必须提供 `15-开发总提示词.md`。

## 5. 长期执行规则

后续所有软件系统开发材料都按本标准生成。

如果用户没有指定目录，默认生成到当前工作目录：

```text
docs/mvp/
```

如果用户指定了参考项目，应先读取参考项目 `docs/mvp` 的结构，再按同结构生成新项目材料。

## 6. 大段提示词交付规则

以后凡是需要交付给 AI 开发平台、外部开发团队或其他执行方的大段提示词，默认生成 Word 文档版本（`.docx`），并保存到当前工作目录。

执行要求：

- 聊天中只给出简短说明和 Word 文件路径，不默认粘贴完整长提示词。
- Word 文档应包含“使用说明”和“正式提示词”两部分。
- 正式提示词必须可整体复制给执行平台。
- 如果提示词属于某个 MVP 阶段，文件名应包含阶段名称，例如 `MVP-1开发提示词.docx`。
- 如用户明确要求纯文本，再额外输出纯文本版本。
